## What this is

`whoami` is the personal portfolio / CV website of Seilbek Skindirov (GitHub
[`prorochestvo`](https://github.com/prorochestvo), repo `github.com/prorochestvo/whoami`).
It is generated by a **Go static-site generator that runs at BUILD TIME** ‚Äî there is no
live server. `cmd/whoami` reads CV content, fetches GitHub stats, renders everything
through `html/template`, and bakes static HTML plus assets into `build/`, which is
deployed to Cloudflare Pages. The aesthetic is code-editor / terminal: self-hosted
JetBrains Mono, a dark theme, near-monochrome with a single accent color.

## Build & Run Commands

Pure Go, `CGO_ENABLED=0`. The generator produces the deployed artifact under `build/`.

```bash
make build    # CGO_ENABLED=0; generator -> build/
make run      # build, then serve build/ locally
make test     # go fmt + vet + test ./...
make format   # go fmt ./...
make clean    # rm build/ + go mod tidy
```

Targeted runs: `CGO_ENABLED=0 go test -run 'TestName[/subtest_name]' ./internal/...` (add `-v` for a single package).

Do not invent `make` targets beyond the five above.

## Architecture Overview

The layout follows a DDD-style split (mirroring the `beacon` project):
`domain` ‚Üí `repository` / `infrastructure` ‚Üí `application` ‚Üí `dto`, wired by `cmd`.

Build-time generation flow:

```
repository/resume (CV data) ‚îÄ‚îÄ‚îÄ‚îê
                               ‚îú‚îÄ‚Üí application/site ‚Üí html/template ‚Üí build/ ‚Üí Cloudflare Pages
infrastructure/github (stats) ‚îÄ‚îò   (Builder ‚Üí dto.Page ‚Üí Renderer; + copy web/* into build/)
```

`cmd/whoami` is the composition root: it wires the r√©sum√© repository, the GitHub client,
and the renderer, then runs the build. Stats are fetched once at build time; they refresh
in production via a scheduled rebuild (cron ‚Üí deploy hook), never by a live server.

| Layer | Location | Role |
|-------|----------|------|
| Entry point | `cmd/whoami/main.go` | Composition root: reads env, wires repository + GitHub client + renderer, runs the build |
| Domain | `internal/domain/` | Pure CV value objects (one type per `.go` file). No dependencies. |
| Repository | `internal/repository/resume/` | In-memory source of truth for the CV content; `Load()` returns a `domain.Resume`. |
| Infrastructure | `internal/infrastructure/github/` | Build-time GitHub API client ‚Üí `domain.GitHubStats`. MUST degrade gracefully (offline / no token / rate-limited) so the build never fails. |
| Application | `internal/application/site/` | `Builder` (use-case: fetch stats, assemble `dto.Page`) and `Renderer` (html/template ‚Üí `build/`, copy `web/*`). Defines the `StatsFetcher` interface it consumes. |
| DTO | `internal/dto/` | `Page` view model passed to templates (flattened r√©sum√© + stats + SEO fields). |
| Templates | `templates/*.tmpl` | `html/template` source (`layout` + `content`). |
| Web | `web/` | Static source: `css/`, `js/`, `fonts/` (`jetbrains-mono-{400,500,700}.woff2`), `img/`, `robots.txt`, `_headers`. |
| Build | `build/` | Generated output. Gitignored. The Cloudflare Pages output directory. |

**Dependency direction (inward):** `cmd` wires everything; `application/site` -> `domain`+`dto` behind a `StatsFetcher` interface; `infrastructure/github` and `repository/resume` -> `domain` only; `domain` -> nothing. `StatsFetcher` lets a failed/empty fetch degrade gracefully and be faked in tests.

## Deployment (Cloudflare Pages)

No server runs anywhere ‚Äî the static `build/` directory is served by Cloudflare Pages. Full setup: [`docs/deployment.md`](docs/deployment.md).

GitHub Actions is the primary path (self-contained, pins its own Go version): `ci.yml` validates PRs (`gofmt`, `go vet`, `go test`, generator build); `deploy.yml` runs the generator + `wrangler pages deploy build` on push to `main`, a monthly `schedule`, and manual dispatch ‚Äî the scheduled run refreshes baked-in GitHub stats with no code change. Secrets `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID`, variable `WHOAMI_SITE_URL`; the stats fetch's built-in `GITHUB_TOKEN` is rate-limit only ‚Äî never in output, never committed.

`web/_headers` (copied into `build/`) defines caching + security headers (CSP, `X-Content-Type-Options`).

## Conventions

- **SEO and link previews are the product.** Every page MUST have a meaningful `<title>`,
  `<meta name="description">`, a canonical URL, and OpenGraph + Twitter card tags
  (`og:title`, `og:description`, `og:image`, `og:url`, `twitter:card`). All real content
  is rendered into the HTML at build time ‚Äî never injected client-side ‚Äî so crawlers and
  preview bots see it.
- **Semantic, accessible HTML.** Use `<header>/<main>/<section>/<article>/<footer>`, a
  real heading hierarchy, `alt` text on every image, labels on any control.
- **Progressive enhancement.** The CV must be fully readable with JS disabled. JS is for
  enhancement only, never a prerequisite for seeing content.
- **Auto-escaping.** All injected GitHub / API data flows through `html/template`'s
  contextual auto-escaping. Do not bypass it (`template.HTML` etc.) for external data.
- **Secrets via env only.** Never read or edit `.env`.
- **Go and generic conventions** (style, declaration order, test structure, godoc,
  error discipline, code organization) come from the `stack-go` plugin skills ‚Äî not
  restated here.

## Constraints

- **Pure Go, `CGO_ENABLED=0`** for all build and test commands.
- **The generated front-end is vanilla HTML / CSS / JS** ‚Äî no framework, bundler, npm, or WASM (WASM was rejected because it kills SEO: content would not be in the HTML).
- **Fonts are self-hosted** (`web/fonts/*.woff2`) ‚Äî no runtime CDN.
- **The GitHub token never appears in generated output** and is never committed.
- **The GitHub fetch must never break the build.** Offline, no token, or rate-limited ‚Üí
  fall back gracefully and render the site without (or with cached) stats.
- **Never run `go build` without `-o ./build/...`.** `build/` (output), `tmp/` (scratch), `logs/` are gitignored.

## Working agreement

All non-trivial work follows the plan-first pipeline:

1. **Plan** ‚Äî the `architect` agent writes `plans/NNN-slug.md` (create via the
   `pipeline:new-plan` skill). No source edits before a plan exists.
2. **Implement** ‚Äî the `engineer` agent executes the plan's tasks with tests.
3. **Review** ‚Äî three `reviewer` agents launched in parallel in ONE message, each
   prompt naming its lens and the changed files. **Project lens override ‚Äî lens B is
   SEO, accessibility & security**: presence/correctness of `<title>`, meta
   description, canonical URL, OG + Twitter tags; semantic structure and heading
   hierarchy; `alt` text; content readable with JS disabled; no auto-escaping bypass
   (`template.HTML`) on external data; `_headers` (CSP) and `robots.txt` sanity.
   Lenses A (correctness & tests) and C (performance & architecture) are standard.
   Full three-lens fan-out is mandatory on the first review; the post-fix re-review
   is ONE solo reviewer scoped to the changed lines.
4. **Gate** ‚Äî `make test` (and `make build` for generator/template changes) must be
   green before review; a red tree goes to the `testdoctor` agent first.
5. **Complete** ‚Äî the orchestrator merges the three reports, deduplicates, resolves
   conflicting verdicts (naming what was rejected and why; the user has final say).
   P0/P1 findings loop back to the engineer. Only when every P0/P1 is fixed or
   explicitly accepted: move the plan via the `pipeline:complete-plan` skill.

Plans live in `plans/` (active), `plans/completed/` (shipped, `YYMMDD.NNNN.slug.md`),
`plans/history/` (abandoned/superseded). One plan per concern.
