# Task Breakdown

## Overview

Add a per-client rate-limit middleware to the public HTTP API to prevent a single caller from exhausting the database connection pool. Last week's incident showed one misbehaving client (looping on a 4xx) generating 300 RPS against `/v1/quotes`, starving everyone else.

## Assumptions

- Rate-limit key = `X-Api-Key` header. Anonymous (no key) requests are bucketed under the client IP. The auth middleware already runs before this one, so the key is on the request context.
- Token-bucket limiter from `golang.org/x/time/rate` is sufficient — no need for a distributed limiter (single-binary deployment).
- Limits live in config: `RateLimitRPS` (default 20) and `RateLimitBurst` (default 40). One bucket per key, evicted after 10 min of idleness.

## Tasks

### Task 1: `internal/middleware/ratelimit` package
- Description: New package exposing `New(cfg Config) func(http.Handler) http.Handler`. Internally an `lru.Cache[string, *rate.Limiter]` keyed by api-key-or-ip. On limit hit return `429 Too Many Requests` with `Retry-After` header and a `PublicError` JSON body.
- Acceptance Criteria:
  - `TestNew` with subtests: `allows_within_burst`, `rejects_over_burst`, `recovers_after_window`, `429_body_matches_contract`, `retry_after_header_present`.
  - Uses `clockwork.Clock` injection for deterministic time control in tests (no `time.Sleep` in tests).
- Pitfalls & edge cases: Map-based cache without eviction will leak memory. Use the existing `internal/cache/lru` with a TTL. Mutex around the map; benchmark to confirm no contention at 1k RPS.
- Complexity: Medium

### Task 2: Wire into `cmd/api/server.go`
- Description: Insert the middleware in the chain *after* auth (so the key is known) and *before* the handler. Behind a feature flag `RATE_LIMIT_ENABLED` defaulting to `false` so we can dark-deploy.
- Acceptance Criteria:
  - Integration test in `cmd/api/server_test.go` confirms 429 on 50th request when burst=10 and rps=1.
  - Server boots and serves traffic with the flag off (no behavioral change).
- Pitfalls & edge cases: Healthcheck endpoints must be exempt — list path prefixes in the limiter's `Skip` config.
- Complexity: Easy

### Task 3: Metrics
- Description: Counter `api_rate_limit_rejections_total{path}` incremented on each 429. Gauge `api_rate_limit_active_buckets` showing current cache size.
- Acceptance Criteria:
  - Metrics registered on the existing Prometheus registry.
  - Scrape test asserts both metrics appear after a 429.
- Pitfalls & edge cases: Don't label by api-key (cardinality explosion); label by path bucket and a derived `client_kind` (`key|ip`).
- Complexity: Easy

## Execution Order

1. Task 1 (middleware package + unit tests)
2. Task 3 (metrics — easier to add now than later)
3. Task 2 (wiring + integration test)

## Risks

- `golang.org/x/time/rate` burst behavior is per-bucket; clients with bursty traffic patterns may legitimately get 429s. Mitigation: roll out with `RATE_LIMIT_ENABLED=false` first, observe `would_reject` counter for 48h, then flip.
- LRU eviction during a request can lose a bucket and reset a client's window. Mitigation: TTL ≥ 10 × token-refill window; acceptable.

## Trade-offs

- In-process limiter doesn't survive restarts and doesn't coordinate across replicas. Fine for the current single-binary deployment; revisit with Redis-backed limiter if we go multi-replica.
- Burst=40 means a client can spike to 40 in the first instant. Lower would be safer but rejects legitimate retry-storms; reviewer to weigh in.
