# Task Breakdown

## Overview

Refactor the `whoami` static-site generator so that CV content is sourced from
per-locale JSON files (single source of truth) instead of the hardcoded Go literal in
`internal/repository/resume/resume.go`, then build an SEO-safe, i18n-ready site with a
progressively-enhanced client-side language switcher.

The work is split into two independently shippable phases:

- **Phase 1 (no visible change, ships alone):** move `en` content into
  `content/en.json`, load + validate it in the repository layer, derive SEO fields from
  that data, and emit the locale JSON into `build/content/en.json`. The rendered page is
  byte-for-byte equivalent to today (modulo the new `<link rel="alternate">` /
  `connect-src` plumbing, which is inert with a single locale). Fully shippable.

- **Phase 2 (becomes user-visible once a 2nd locale exists):** loop over all locales,
  pre-render a full static page per locale (`build/index.html` for `en`,
  `build/<lang>/index.html` for the rest), emit correct `hreflang` cross-links +
  `x-default`, turn the `lang=en` titlebar indicator into a real `<select>`/link control
  that works with no JS, and layer on the JS fetch-and-swap switcher with a CSS progress
  bar that respects `prefers-reduced-motion`. Testable today with `en` plus a
  clearly-marked placeholder second locale.

Affected layers (dependency direction preserved — `domain` depends on nothing,
consumers define their own interfaces, rendering stays in `application`):

| Concern | Layer / file |
|---|---|
| Pure CV value objects | `internal/domain/*.go` |
| Load + validate locale JSON | `internal/repository/resume/resume.go`, new `content/*.json` |
| GitHub stats fetch (unchanged) | `internal/infrastructure/github/client.go` |
| View model + SEO + hreflang set | `internal/dto/page.go` |
| Render per-locale HTML + emit JSON | `internal/application/site/{builder,renderer}.go` |
| Templates | `templates/{layout,index}.html.tmpl` |
| Static enhancement | `web/js/main.js`, `web/css/style.css` |
| Composition root / locale list | `cmd/whoami/main.go` |

## Assumptions

1. **`en` is the default locale.** It renders to `build/index.html` (no `/en/` prefix).
   Every other locale `xx` renders to `build/xx/index.html`. This preserves the existing
   indexed URL (`/`) — crawlers that already know `/` keep working. (Risk: backward
   compatibility of output URLs — see Risks.)
2. **Locale set is config, not magic.** The list of locales to build is owned by the
   composition root (`cmd/whoami/main.go`), defaulting to `["en"]` in Phase 1 and
   `["en", <placeholder>]` for Phase 2 testing. The repository exposes "load locale X"
   and "list available locales"; it does not decide which get rendered.
3. **Content is embedded via `//go:embed content/*.json`** for a hermetic build (no
   runtime disk dependence on CWD, matches the project's embed convention). The
   `content/` directory lives at the module root next to `go.mod`. (Trade-off vs
   read-from-disk discussed in Trade-offs.)
4. **JSON maps to domain via a thin repository-local DTO, NOT by tagging domain
   structs.** `internal/domain` stays a pure vocabulary with zero serialization
   concerns; the repository owns the on-disk schema and the mapping. (Justification in
   Trade-offs.)
5. **The same JSON shape feeds both the build render and the runtime swap.** The file
   emitted to `build/content/<lang>.json` is the *same* per-locale content file, copied
   verbatim — there is exactly one schema, authored once.
6. **No new third-party deps.** Vanilla `encoding/json`, `html/template`, `embed`.
   Switcher is hand-written vanilla JS. No framework/bundler/npm/WASM (CLAUDE.md
   constraint).
7. **CSP currently allows the fetch.** `web/_headers` sets `default-src 'self'` and no
   explicit `connect-src`, so a same-origin `fetch('/content/xx.json')` is already
   permitted. We will add an explicit `connect-src 'self'` to make intent unambiguous
   and survive any future CSP tightening (Task 9).
8. **The placeholder second locale is throwaway.** It exists only to exercise the
   multi-locale code paths and `hreflang` correctness in tests/visual check; it is NOT a
   real translation and must be obviously marked so it is never mistaken for shippable
   content. Real translations are out of scope here.
9. **GitHub stats remain locale-independent and graceful.** Stats are fetched once and
   reused across every locale render; a failed fetch still never fails the build
   (existing contract in `builder.go`).

## Tasks

### Task 1: Define the on-disk JSON schema and author `content/en.json`

- Description:
  Create `content/en.json` at the module root carrying the *exact* current `en` content
  from `internal/repository/resume/resume.go`. The JSON object mirrors the `domain`
  aggregates: top-level keys `person`, `about`, `experience`, `education`, `skills`,
  `languages`, `relocation`, `contacts`. Field names use lower-snake or lowerCamel —
  pick one and apply consistently across the file; document the choice in a top-of-repo
  comment in `resume.go` (Task 2) since JSON can't carry comments. Add a `"locale": "en"`
  and `"localeName": "English"` header field so the switcher and `<html lang>` can read a
  self-describing label without a separate registry.
  This is content authoring only — no Go changes in this task.
- Acceptance Criteria:
  - `content/en.json` parses as valid JSON (`python3 -m json.tool content/en.json` or
    `go run` smoke).
  - Every string currently in `resume.go` (all 8 experience entries, every highlight,
    every stack tag, contacts incl. empty-URL Location, both spoken languages, relocation
    list) is present verbatim — no dropped or reordered entries.
  - The single `"current": true` experience flag is set on the Yoxel entry and nowhere
    else.
  - `person` includes `name`, `title`, `location`, `tagline`, `birthYear` (number),
    `experience` (string e.g. `"17+ years"`), `githubUser`.
  - File includes `"locale": "en"` and `"localeName": "English"`.
- Pitfalls & edge cases:
  - `birthYear` is an `int` in `domain.Person` — keep it a JSON number, not a string, or
    unmarshalling fails.
  - Contacts with empty `URL` must serialize the empty string (or omit and let Go
    zero-value it) — the template branches on `if .URL`, so the round-trip must preserve
    "empty means plain text".
  - Email casing (`SeilbekSkindirov@gmail.com`) and the `mailto:` URL must match exactly;
    `main.js` matches on `a[href^="mailto:"]`.
  - Do not let an editor reorder JSON keys in a way that reorders *array* elements —
    object key order is irrelevant but array order is load-bearing (it's render order).
- Complexity: Easy

### Task 2: Repository loads + validates a locale's JSON into `domain.Resume`

- Description:
  Rewrite `internal/repository/resume/resume.go` so the content comes from embedded JSON,
  not a Go literal. Add `//go:embed content/*.json` (the directive must reference a path
  relative to the Go file; place the embed in a file whose package can see `content/` —
  if `content/` is at module root and the package is `internal/repository/resume`, embed
  from a small loader at module root or move `content/` under the repository package.
  **Decision: put `content/` under `internal/repository/resume/content/`** so the embed
  directive is local to the owning layer and the path is unambiguous; update Task 1's
  path accordingly). Introduce an unexported `resumeJSON` DTO struct in the repository
  package with `json` tags matching Task 1's schema, plus per-section sub-structs, and a
  mapping function `toDomain(resumeJSON) domain.Resume`. Replace `Load()` with:
  - `Load(locale string) (domain.Resume, error)` — unmarshal `<locale>.json`, validate,
    map to domain.
  - `Available() []string` — list embedded locale codes (filename stems), sorted.
  Validation rejects structurally-broken content: empty `person.name`, empty
  `person.githubUser`, zero experience entries, and **not exactly one** `current:true`
  experience (preserves the invariant the existing test asserts).
- Acceptance Criteria:
  - `domain` package is untouched — no `json` tags added there (`git diff
    internal/domain` is empty).
  - `Load("en")` returns a `domain.Resume` deep-equal to the previous hardcoded value
    (assert key fields; ideally snapshot the whole struct).
  - `Load("does-not-exist")` returns a non-nil error, not a panic.
  - `Available()` returns `["en"]` (Phase 1) sorted ascending.
  - Validation: a fixture with two `current:true` entries returns a `PublicError`-style
    or plain error and is covered by a test; a fixture with empty `person.name` errors.
  - `resume_test.go` updated: `TestLoad` becomes `TestLoad` with subtests
    `loads en content`, `errors on unknown locale`, `errors on invalid content (no
    current role)`, `errors on invalid content (multiple current roles)`. One top-level
    test per function per CLAUDE.md.
  - `TestAvailable` covers the locale listing.
- Pitfalls & edge cases:
  - `//go:embed` path is relative to the `.go` file and cannot use `..` — this is the
    reason for co-locating `content/` under the package.
  - `embed.FS` reads are deterministic but `fs.ReadDir` order is sorted already; still
    sort explicitly so the contract is documented, not incidental.
  - Validation must run *after* unmarshal — a malformed JSON file (syntax error) should
    surface a distinct, wrapped error (`fmt.Errorf("resume: parse %s: %w", ...)`) so the
    build log points at the offending locale.
  - Do not leak the GitHub token anywhere near here — this layer never sees it; just
    don't add logging that echoes arbitrary content fields (none contain secrets, but
    keep error messages to the locale name + parse error, not full file dumps).
  - Keep `Load` returning a value + error; callers (`main.go`) must `log.Fatalf` on a
    bad *default* locale (a broken `en` is a build-breaking error, unlike a graceful
    GitHub miss).
- Complexity: Medium

### Task 3: Emit each locale's JSON into the build output

- Description:
  Teach the renderer (or builder) to copy each built locale's source JSON to
  `build/content/<lang>.json` so the client switcher can `fetch` it. Add a
  `Renderer.WriteLocaleJSON(locale string, raw []byte)` method (or have the builder pass
  the already-loaded raw bytes). Source of the bytes: the same embedded file the
  repository read — expose it via a new repository function
  `Raw(locale string) ([]byte, error)` returning the embedded JSON unmodified, so build
  render and runtime swap share one byte source (Assumption 5). Write into
  `build/content/` with `0o644`, creating the dir.
- Acceptance Criteria:
  - After a build, `build/content/en.json` exists and is byte-identical to the embedded
    `en.json`.
  - The file is served as `application/json` by Cloudflare Pages by extension (no code
    needed, but verify `_headers` doesn't accidentally block `/content/*` — it doesn't;
    `default-src 'self'` covers it).
  - Renderer test asserts the JSON file is written and matches the input bytes.
  - No secret/token can appear: the bytes written are exactly the embedded content file,
    which contains no token (assert in a test that the emitted JSON does not contain the
    substring of any env-derived token — practically, assert it equals the source).
- Pitfalls & edge cases:
  - Don't re-marshal `domain.Resume` to produce the JSON — that would risk drift between
    the build-render shape and the swap shape, and would strip the `locale`/`localeName`
    header fields. Copy the *source* bytes.
  - `build/content/` must be created before write; the existing `copyAssets` walk only
    mirrors `web/`, so this is a separate write step ordered after `MkdirAll(outDir)`.
  - Cache busting: the switcher will fetch `/content/<lang>.json` — decide whether to
    append `?v=<buildstamp>` at fetch time (client-side) rather than renaming the file,
    so the on-disk name stays stable and predictable.
- Complexity: Easy

### Task 4: Move SEO derivation + per-locale URL/hreflang into the DTO

- Description:
  Extend `internal/dto/page.go` so the view model carries everything a per-locale page
  needs for SEO. Add to `Page`:
  - `Lang string` (BCP-47 code for `<html lang>`),
  - `LocaleName string` (human label for the switcher current value),
  - `Alternates []Alternate` where `Alternate{Lang, Href}` lists every locale's absolute
    URL plus an `x-default` entry, for `<link rel="alternate" hreflang>`,
  - `LocaleLinks []LocaleLink` (`{Lang, Name, Href, Current bool}`) to render the no-JS
    switcher control.
  Change `NewPage` to `NewPage(r domain.Resume, s domain.GitHubStats, siteURL, lang
  string, allLocales []LocaleMeta, generatedAt time.Time)`. `LocaleMeta{Code, Name}` is a
  small dto/value type describing the full locale set so the function can compute every
  alternate and the per-locale canonical. Canonical/href rule: `en` → `siteURL`, `xx` →
  `siteURL + "xx/"`. OG `og:locale` reflects `lang`.
- Acceptance Criteria:
  - `NewPage(..., "en", [{en,English}], ...)` yields `CanonicalURL == siteURL`,
    `Alternates` contains `{en, siteURL}` and `{x-default, siteURL}`.
  - With two locales, `NewPage(..., "de", [{en,English},{de,Deutsch}], ...)` yields
    `CanonicalURL == siteURL+"de/"`, and `Alternates` contains all of:
    `{en, siteURL}`, `{de, siteURL+"de/"}`, `{x-default, siteURL}` (x-default points at
    the default locale).
  - `Lang`/`LocaleName` reflect the rendered locale.
  - Existing SEO behavior preserved: `Title == name + " — " + title`,
    `Description == tagline`, trailing-slash normalization on `siteURL` still applies.
  - `page_test.go` extended: subtests `derives SEO fields`, `default locale canonical has
    no prefix`, `non-default locale canonical is prefixed`, `alternates include
    x-default pointing at default`.
- Pitfalls & edge cases:
  - `x-default` must point at the default locale's URL (`siteURL`), per Google's
    guidance — getting this wrong is the classic hreflang bug.
  - hreflang values must be the same codes used in URLs and `<html lang>`; a mismatch
    (e.g. `en-US` in the tag but `/en/` in the path) is invalid.
  - All alternates must be **absolute** URLs (scheme+host), not path-relative — relative
    hreflang is ignored by crawlers.
  - Keep `og:locale` format `xx_XX` only if you actually have region; for bare language
    codes, `lang` attr stays `en`/`de` and `og:locale` can be omitted rather than
    emitting a malformed value.
- Complexity: Medium

### Task 5: Render one full static page per locale

- Description:
  Convert the single-page render into a locale loop. The builder (`builder.go`) gains the
  locale set and, for each locale, loads the resume (Task 2), builds a `Page` (Task 4),
  and asks the renderer to write it to the correct path. Update `Renderer.Render` (or add
  `RenderLocale(page dto.Page, outRelDir string)`) so it writes `index.html` into
  `outDir` for the default locale and `outDir/<lang>/` for others. Assets (`web/`) are
  copied **once**, not per locale. `NewBuilder` signature changes from a single `resume`
  to a way to load per-locale content: pass the repository's `Load`/`Available`/`Raw` in
  as a small interface defined in `site` (consumer-defined interface, per DDD rule), e.g.
  `type ContentSource interface { Available() []string; Load(string) (domain.Resume,
  error); Raw(string) ([]byte, error) }`.
- Acceptance Criteria:
  - With locales `["en"]`: `build/index.html` exists; no `build/en/` directory is
    created (default locale is not duplicated).
  - With locales `["en","de"]`: both `build/index.html` and `build/de/index.html` exist,
    each containing that locale's content.
  - `build/content/en.json` (and `de.json`) emitted (Task 3) for every built locale.
  - Assets copied once: `build/css/style.css` exists and `copyAssets` is invoked a single
    time regardless of locale count (assert via a counter or by checking it's outside the
    loop).
  - GitHub stats fetched once and identical across locales (the fetch is not repeated per
    locale).
  - A failed GitHub fetch still produces every locale page (graceful contract holds
    across the loop) — extend `builder_test.go` to assert both locales render on fetch
    error.
  - `builder_test.go` updated with a fake `ContentSource` returning two locales.
- Pitfalls & edge cases:
  - Don't fetch GitHub stats inside the locale loop — fetch once before it (perf + the
    20s context budget in `main.go` is shared).
  - Path join for nested locale dir must `MkdirAll(outDir/<lang>)` before
    `os.Create`.
  - The default-locale special-case (no prefix) must match the canonical-URL rule in
    Task 4 exactly, or the page's own canonical will disagree with where the file lives.
  - Template parse happens once in `NewRenderer`; reusing the parsed template across
    locales is fine (templates are stateless per execute) — don't re-parse per locale.
  - Keep `Build` returning an error only for output-invalidating failures (a broken
    *non-default* locale: decide policy — recommend fail the whole build so you never
    ship a half-translated site with dangling hreflang; document this).
- Complexity: Hard

### Task 8: Progressive-enhancement JS switcher with fetch-and-swap

- Description:
  Extend `web/js/main.js` (still IIFE, vanilla, no deps) so that when JS is available it
  upgrades the no-JS link control: intercept activation of a locale link, show the CSS
  progress bar (Task 9), `fetch('/content/<lang>.json?v=<stamp>')`, and swap the rendered
  DOM regions in place — name (`#name`), role, tagline, facts/contacts, about,
  experience, skills, education, languages — then update `<html lang>`,
  `document.title`, the `<meta name="description">`, OG/Twitter title+description, the
  `<link rel="canonical">`, and the URL via `history.pushState`. On any fetch/parse
  error, fall back to normal navigation (`window.location = href`). Wire `popstate` so
  back/forward re-swaps (or simply reloads). The DOM-building must mirror the template's
  structure so swapped content is identical to a server-rendered visit.
- Acceptance Criteria:
  - With JS on: activating a locale swaps content with no full reload, updates
    `document.title`, `<html lang>`, `<meta name=description>`, canonical, and the address
    bar to the locale URL.
  - With JS on but `fetch` failing (simulate 404/offline): the click falls through to a
    normal navigation to the pre-rendered page (no broken state, no console-only failure).
  - `history.pushState` + `popstate` keep the URL and content in sync on back/forward.
  - The email click-to-copy behavior (existing) still works after a swap (re-bind
    handlers on swapped DOM, or use event delegation so it survives swaps).
  - Swapped DOM is structurally equal to the corresponding pre-rendered page for that
    locale (spot-check a couple of regions in a test or manual diff).
  - All values inserted into the DOM use `textContent` / `createElement`, never
    `innerHTML` with fetched strings (XSS parity with `html/template` escaping).
- Pitfalls & edge cases:
  - **XSS:** the JSON is your own content, but treat it as untrusted in the DOM layer —
    `textContent` only; never `el.innerHTML = data.x`. This mirrors the build-time
    auto-escaping invariant.
  - CSP `script-src 'self'` means no inline JS and no `eval`/`new Function` — keep
    everything in the external file; `JSON.parse` is fine.
  - CSP `connect-src` must allow the fetch — add `connect-src 'self'` in Task 9.
  - URL/state desync: update `<html lang>` and canonical together with `pushState`, or
    the back button leaves stale meta.
  - Re-binding handlers after swap: prefer event delegation on a stable ancestor so the
    mailto-copy and future handlers survive content replacement.
  - Don't refetch a locale already loaded — cache parsed JSON in a `Map` to keep
    switching instant.
  - `prefers-reduced-motion`: the progress bar must not animate when the user opts out
    (handled in CSS, Task 9 — JS still toggles the class but the animation is suppressed).
- Complexity: Hard

    `<xhtml:link rel="alternate" hreflang="…">` for all locales + `x-default`, absolute
    URLs.
  - `robots.txt` (now possibly templated or appended) contains a `Sitemap:
    <siteURL>sitemap.xml` line.
  - With a single locale, the sitemap is still valid (one `<url>`).
  - Renderer test asserts sitemap contains both locale URLs in a 2-locale build.
- Pitfalls & edge cases:
  - Sitemap hreflang must agree with the `<link rel=alternate>` in the pages — derive
    both from the same Task-4 data to avoid drift.
  - `robots.txt` currently lives in `web/` and is copied verbatim; if you template the
    `Sitemap:` line you must generate it into `build/` rather than copying — pick one
    approach (recommend: keep `web/robots.txt` static with a hard-coded `Sitemap:` line
    pointing at the canonical host, since `siteURL` for prod is known and stable).
  - Absolute URLs again — relative sitemap entries are ignored.
- Complexity: Medium

## Risks

- **Output-URL backward compatibility.** Default locale must stay at `/` (not `/en/`) so
  any crawler-indexed `/` URL and existing share links keep resolving. Mitigated by
  Assumption 1 + Task 4/5 default-locale special-case. Changing this later would orphan
  indexed URLs.
- **hreflang correctness is easy to get subtly wrong** (relative URLs, missing
  `x-default`, code/path mismatch). A wrong hreflang is worse than none — it can suppress
  the wrong page in search. Mitigated by deriving all hreflang from one DTO source
  (Task 4) and asserting in tests (Task 4, 11).
- **CSP blocking the fetch.** If `connect-src` were ever tightened without `'self'`, the
  switcher silently fails (it falls back to navigation, so not catastrophic, but the
  enhancement dies). Task 9 makes `connect-src 'self'` explicit.
- **DOM-swap divergence.** The JS-built DOM can drift from the template output over time,
  so a swapped page differs from a directly-loaded one. Mitigated by structural-equality
  spot checks (Task 8) and by keeping the JSON schema the single source. This is the main
  long-term maintenance cost of the swap approach.
- **Placeholder locale leaking to production.** A throwaway locale could ship if
  `WHOAMI_LOCALES` is misconfigured. Mitigated by defaulting to `en` and visible marker
  text (Task 10).
- **Token leakage.** Lower risk here (content layer never sees the token), but Task 3
  must copy *content* bytes only, and error messages (Task 2) must not dump environment.
  Existing graceful-fetch contract for GitHub is preserved across the locale loop (Task
  5).
- **Build determinism.** `Available()` and alternates must be sorted/ordered
  deterministically so the generated HTML/sitemap are byte-stable across builds (clean
  diffs, cache-friendly). Tasks 2, 4, 11 specify explicit sorting.

## Trade-offs

- **Embed vs read-from-disk for content.** Chose `//go:embed` (Assumption 3) for a
  hermetic build that doesn't depend on CWD and matches the project's embed convention.
  Cost: editing content requires a rebuild (fine — the whole site is built anyway) and
  the binary carries the JSON. Reading from disk would allow content edits without
  recompiling but introduces a runtime path dependency and a failure mode (missing dir)
  the embed approach eliminates. Embed wins for a build-time generator.
- **Thin repository DTO vs `json`-tagging domain structs.** Chose a repository-local
  `resumeJSON` DTO + `toDomain` mapper (Assumption 4). Cost: a little mapping boilerplate
  and keeping two shapes in sync. Benefit: `internal/domain` stays a pure, dependency-free
  vocabulary (CLAUDE.md layer rule — domain has no logic and no serialization concern),
  and the on-disk schema can evolve (rename keys, add `locale`/`localeName` header) without
  touching domain or rippling into the DTO/template layers. Tagging domain would couple
  the wire format to the domain model forever.
- **Pre-render every locale vs render `en` + client-only translations.** Chose full
  per-locale pre-render (the agreed architecture) — non-negotiable for SEO: crawlers and
  link-preview bots see real, complete content per language without executing JS. Cost:
  build time and output size scale linearly with locale count, and N copies of largely
  identical markup. Acceptable for a CV with a handful of locales.
- **Links-baseline + JS-upgraded `<select>` vs `<select>`-first.** Chose plain `<a>`
  links as the crawlable, no-JS-correct baseline (Task 6/7), upgraded by JS. Cost: the
  default control is a link list, not a native dropdown, until JS loads. Benefit: it is
  correct and navigable with zero JS and fully crawlable — which is the whole point.
- **Fetch-and-swap vs full navigation on switch.** Chose fetch-and-swap with graceful
  fallback (agreed). Cost: DOM-build code that must track the template (divergence risk
  above). Benefit: instant, no-reload locale switching with a progress bar, while the
  no-JS path and crawlers still get full navigations to real pages.
- **Sitemap generated vs hand-maintained.** Chose generated-from-locale-set (Task 11) so
  it can never drift from the actual built pages. Cost: a bit more renderer code. Benefit:
  one source of truth for which URLs exist and their hreflang.
